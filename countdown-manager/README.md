# Countdown Manager

A client-side web application built with [Rust](https://www.rust-lang.org/) and
[Leptos](https://leptos.dev/) that provides a browser UI for managing countdown
dates through the `dates-api` service.

---

## Architecture Overview

```
Browser (WASM)                    Host Server
┌─────────────────────────┐       ┌──────────────────────────┐
│  countdown-manager      │       │  dates-api               │
│  (Leptos CSR / Trunk)   │──────▶│  (wasmtime serve)        │
│  port 8000              │ /api/ │  port 8080               │
└─────────────────────────┘       └──────────────────────────┘
                                          │
                                   reads/writes
                                          │
                                  ┌───────────────┐
                                  │  dates/        │
                                  │  YYYYMMDD.txt  │
                                  └───────────────┘
```

The frontend compiles to WebAssembly and runs entirely in the browser
(Client-Side Rendering / CSR). During development, `trunk serve` proxies
`/api/` requests to the `dates-api` backend so CORS is never an issue.
In production, the same proxy pattern is maintained by whatever reverse-proxy
(nginx, Caddy, etc.) sits in front of both services.

### Why CSR + Trunk?

| Concern | Choice | Reason |
|---|---|---|
| Rendering strategy | CSR (WASM in browser) | The backend API already exists; no need for an SSR server. |
| Build tool | [Trunk](https://trunkrs.dev/) | Zero-config, handles WASM, asset bundling, dev proxy, and hot reload. |
| CSS framework | [Tailwind CSS v4](https://tailwindcss.com/) | Utility-first, zero dead CSS (scanner finds classes in Rust source), `@theme` tokens replace a design-system CSS file. |
| HTTP client | `gloo-net` | The standard browser-safe HTTP crate for Rust/WASM. |
| Routing | `leptos_router` | Enables multi-page navigation without a page refresh. |

## Getting Started

### Prerequisites

```bash
# Install the WASM target
rustup target add wasm32-unknown-unknown

# Install Trunk
cargo install trunk

# Install Node dependencies (Tailwind CSS)
npm install
```

### Development

```bash
# Start dates-api on port 19697 first (from the repo root):
wasmtime serve --addr 127.0.0.1:19697 -S cli \
  --dir ./dates::/dates \
  dates-api/target/wasm32-wasip2/release/dates_api.wasm

# Then, in this directory — Trunk runs the Tailwind build step automatically:
trunk serve --port 8000 --open
```

For live CSS changes without a full WASM recompile, run Tailwind in watch
mode in a second terminal:

```bash
npm run css:watch
```

Trunk watches for source changes, recompiles to WASM, and hot-reloads the
browser automatically. The `/api/` proxy in `Trunk.toml` forwards all API
calls to the running `dates-api` instance.

### Production Build

```bash
trunk build --release
# Output lands in dist/
```

## Deploying to Apache

The app is served as **static files** by Apache. Apache also reverse-proxies
`/api/` requests to the `dates-api` `wasmtime serve` process running on the
same host. No Node.js, no SSR server — just a `.wasm` bundle + static assets.

```
Browser ──HTTPS──▶ Apache :443
                    │  GET /            → serves dist/index.html
                    │  GET /*.wasm|js|css → serves dist/pkg/…
                    └  /api/*           → proxy → wasmtime :19697
```

### 1. Build the WASM bundle

```bash
cd countdown-manager/
npm install          # install tailwindcss (first time only)
trunk build --release
# Output: countdown-manager/dist/
```

### 2. Deploy static files

Copy `dist/` to the server path referenced in `deploy/apache.conf`
(default: `/var/www/countdown-manager/dist`):

```bash
rsync -av --delete dist/ user@server:/var/www/countdown-manager/dist/
```

### 3. Install and start dates-api as a systemd service

The `deploy/dates-api.service` unit keeps `wasmtime serve` running and
restarts it automatically on failure. Edit `User`, `Group`, and
`WorkingDirectory` to match your server layout, then:

```bash
sudo cp deploy/dates-api.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now dates-api
sudo systemctl status dates-api   # confirm it's running on 127.0.0.1:19697
```

### 4. Configure Apache

Enable the required modules (Debian/Ubuntu):

```bash
sudo a2enmod proxy proxy_http headers rewrite ssl
```

Install the site config:

```bash
sudo cp deploy/apache.conf /etc/apache2/sites-available/countdown-manager.conf
# Edit ServerName, DocumentRoot, and SSL cert paths first:
sudo nano /etc/apache2/sites-available/countdown-manager.conf

sudo a2ensite countdown-manager
sudo apache2ctl configtest   # must say "Syntax OK"
sudo systemctl reload apache2
```

### Files in `deploy/`

| File | Purpose |
|---|---|
| `apache.conf` | VirtualHost — serves `dist/`, proxies `/api/` → `:19697` |
| `dates-api.service` | systemd unit — runs `wasmtime serve` on loopback port 19697 |

## API Contract

The app talks to the `dates-api` over its three REST endpoints:

| Method | Path | Used by |
|--------|------|---------|
| `GET` | `/api/dates` | `DateList` — load all dates on mount and after mutations |
| `POST` | `/api/dates/{YYYYMMDD}` | `AddDateForm` — request body is the plain-text label |
| `DELETE` | `/api/dates/{YYYYMMDD}` | `DateCard` — fired by the Delete button |

The response shape for `GET /api/dates` is a JSON array:

```json
[
  { "key": "20261225", "label": "Christmas 2026" },
  { "key": "20260704", "label": "Independence Day" }
]
```

This maps directly to the `DateEntry` struct in `src/lib.rs`.

## Design Decisions

### Reactivity model

Leptos 0.8 uses fine-grained reactivity (signals) rather than virtual DOM
diffing, so only the DOM nodes that depend on a changed signal are updated.
A single `refresh` counter signal acts as the "invalidation token": after any
mutation the counter increments, which triggers the `LocalResource` to
re-fetch the full date list from the API.

### No global state manager

For an app this size, component-local signals and prop drilling are sufficient.
If the component tree grows, the natural next step is Leptos
[context](https://docs.rs/leptos/latest/leptos/context/index.html) to share
the `refresh` signal and mutation helpers without threading props through every
layer.

### Optimistic UI (future)

Currently the list re-fetches from the server after every mutation. A future
optimisation is to apply changes to local signal state immediately
(optimistic update) and roll back on error — this would eliminate the
perceptible delay on slow connections.
